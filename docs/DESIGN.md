# Smart Flashcards — Design & Development Plan

## What we're building

A web app where a user signs in with a username/password, uploads a PDF, gives it a
name/description, and gets back an Anki-style flashcard deck generated by Claude. Two tabs:

1. **Upload** — drop a PDF, name the deck, kick off generation.
2. **Decks** — browse decks, flip through cards (study mode), edit/delete cards and decks.

MVP philosophy: everything file-based (no database, no external auth provider), synchronous
generation, minimal dependencies. Upgrade paths are noted where they matter.

## Architecture

```
┌─────────────────┐        ┌──────────────────────┐        ┌─────────────────┐
│  React (Vite)   │  HTTP  │  FastAPI backend      │  SDK   │  Anthropic API  │
│  frontend/      │ ─────► │  backend/             │ ─────► │  (Claude)       │
│  2 tabs: Upload │        │  auth + decks + gen   │        │  PDF → cards    │
│  & Decks        │        │  file storage (data/) │        │  structured out │
└─────────────────┘        └──────────────────────┘        └─────────────────┘
```

- **Frontend**: React + TypeScript via Vite. Dev server proxies `/api` to the backend.
- **Backend**: FastAPI (Python 3.12), managed with **Poetry**. Serves a JSON API under `/api`.
- **Storage**: flat files under `data/` (gitignored). No DB for the MVP.
- **AI**: the official `anthropic` Python SDK, called directly from the backend.

### Why direct Anthropic API instead of Claude Code as the agent backend

The task (PDF in → flashcards out) is a **single LLM call**, not an agent loop — there are no
tools to run, no multi-step exploration. Driving Claude Code headless would add process
management, session handling, and unpredictable token usage for no benefit. Instead:

- The backend sends the PDF **natively** as a `document` content block (Claude reads PDFs
  directly — no client-side text extraction needed; limits: 32 MB, 600 pages).
- Output is enforced with **structured outputs** (`client.messages.parse()` with a Pydantic
  schema), so we always get valid `[{front, back}, ...]` JSON — no parsing failures.
- The *spirit* of the "agent skill" idea is preserved: the generation instructions live in a
  versioned markdown skill file (`backend/app/skills/flashcard_generation.md`) that is loaded
  into the system prompt. Tuning card quality = editing that file, no code changes.

If we later want agentic behavior (e.g. Claude deciding to split a textbook into multiple
decks, or web-augmenting cards), the upgrade path is the **Claude Agent SDK** — noted in
Phase 5, not needed for MVP.

### Model choice

| Model | Price (in/out per MTok) | Notes |
|---|---|---|
| `claude-sonnet-5` | $2 / $10 intro (through 2026-08-31), then $3/$15 | **MVP default (decided)** — newest Sonnet and currently the *cheapest* one thanks to intro pricing |
| `claude-haiku-4-5` | $1 / $5 | Cost floor; fine for straightforward material — worth A/B testing |
| `claude-sonnet-4-5` | $3 / $15 | Older Sonnet; costs more than Sonnet 5's intro pricing, no reason to prefer it |

Model is configured via the `ANTHROPIC_MODEL` env var (default `claude-sonnet-5`) so swapping
is a config change.

A rough cost estimate: a 50-page PDF ≈ 50–100K input tokens ≈ **$0.10–0.25 per deck** on
Sonnet 5 intro pricing.

## Data model

```jsonc
// data/users.json
{ "users": [ { "id": "u_...", "username": "cole", "password_hash": "...", "created_at": "..." } ] }

// data/decks/<user_id>/<deck_id>.json
{
  "id": "d_...",
  "name": "Bio 101 – Chapter 4",
  "description": "Cell respiration",
  "created_at": "...",
  "source_filename": "chapter4.pdf",
  "cards": [
    { "id": "c_...", "front": "What is glycolysis?", "back": "...", "tags": ["metabolism"] }
  ]
}
```

```jsonc
// data/topics/<user_id>/<topic_id>.json   (+ sibling <topic_id>.pdf = original upload)
{
  "id": "t_...",
  "name": "The Water Cycle",
  "description": "...",
  "created_at": "...", "updated_at": "...",
  "source_filename": "water_cycle.pdf",
  "instructions": "...",          // standing guidance to the tutor (focus/question style);
                                   // injected into extraction + every quiz call
  "notes_md": "# ...",            // Claude-extracted study notes; quiz source of truth.
                                   // Dispute corrections append to a "Corrections &
                                   // clarifications" section.
  "mastery_score": 37,             // 0-100, agent-decided at each session end, can decrease
  "mastery_notes": "...",          // the tutor's memory: what's known/untested, why the score
  "sessions": [                    // compact archive; full transcripts are not persisted
    { "id": "q_...", "started_at": "...", "completed_at": "...", "total_questions": 5,
      "questions_answered": 5, "grades": {"good": 3, "ok": 1, "bad": 1},
      "score_before": 30, "score_after": 37, "summary": "..." }
  ],
  "active_session": {              // null when idle; the quiz state machine
    "id": "q_...", "started_at": "...", "total_questions": 5,
    "status": "awaiting_answer",   // -> "awaiting_next" after grading, and back
    "questions": [ { "question": "...", "answer": "...", "grade": "good",
                     "feedback": "...", "disputes": [] } ]
  }
}
```

Writes go through a small `storage/` module (atomic write-then-rename) so a later swap to
SQLite/Postgres only touches one layer.

## API surface

| Method & path | Purpose |
|---|---|
| `POST /api/auth/register` | create user (username + password) |
| `POST /api/auth/login` | verify, set signed session cookie |
| `POST /api/auth/logout` | clear cookie |
| `GET  /api/auth/me` | current user |
| `POST /api/decks` | multipart: `file` (PDF) + `name` + `description` → generates cards, returns deck |
| `GET  /api/decks` | list user's decks (summaries) |
| `GET  /api/decks/{id}` | full deck with cards |
| `PATCH /api/decks/{id}` | rename / edit description |
| `DELETE /api/decks/{id}` | delete deck |
| `POST /api/decks/{id}/cards` | add a card |
| `PATCH /api/decks/{id}/cards/{card_id}` | edit a card |
| `DELETE /api/decks/{id}/cards/{card_id}` | delete a card |
| `POST /api/topics` | multipart: `file` (PDF) + `name` + `description` + `instructions` → extracts notes doc, returns topic |
| `GET  /api/topics` | list topic summaries (mastery score, session count, active-session flag) |
| `GET  /api/topics/{id}` | full topic: notes doc, mastery memory, sessions, active session |
| `PATCH /api/topics/{id}` | rename / edit description / edit instructions |
| `DELETE /api/topics/{id}` | delete topic (+ stored PDF) |
| `GET  /api/topics/{id}/pdf` | download the original PDF |
| `POST /api/topics/{id}/quiz/start` | `{num_questions (1-25), replace}` → first question; 409 if a session is active and `replace` is false |
| `POST /api/topics/{id}/quiz/answer` | `{answer, session_id?, question_number?}` → Good/Ok/Bad grade + feedback |
| `POST /api/topics/{id}/quiz/next` | next adaptive question |
| `POST /api/topics/{id}/quiz/dispute` | `{message, ...binding}` → verdict (can raise the grade, may correct the notes doc) |
| `POST /api/topics/{id}/quiz/finish` | score the session: updates mastery score + memory, archives the session |
| `DELETE /api/topics/{id}/quiz` | abandon the active session without scoring |

`POST /api/decks` is **synchronous** for the MVP (generation takes ~15–60s; the frontend shows
a progress state). Upgrade path: FastAPI `BackgroundTasks` + a deck `status` field
(`generating` → `ready`) with polling.

## Generation pipeline (backend/app/generation/)

```python
class Flashcard(BaseModel):
    front: str
    back: str
    tags: list[str] = []

class FlashcardDeck(BaseModel):
    cards: list[Flashcard]

response = client.messages.parse(
    model=settings.anthropic_model,           # claude-sonnet-5
    max_tokens=32000,   # non-streaming budget; 128k is possible but needs streaming
    system=load_skill("flashcard_generation.md"),   # the versioned skill prompt
    messages=[{
        "role": "user",
        "content": [
            {"type": "document",
             "source": {"type": "base64", "media_type": "application/pdf", "data": pdf_b64}},
            {"type": "text", "text": f"Create flashcards for a deck named '{name}': {description}"},
        ],
    }],
    output_format=FlashcardDeck,
)
deck = response.parsed_output   # validated FlashcardDeck — no JSON parsing failures
```

Validation guardrails: reject non-PDF uploads, enforce a file-size cap (< 32 MB API limit,
practically ~20 MB), and surface `stop_reason == "max_tokens"` as a friendly "document too
large" error.

## Development plan

Each phase = its own worktree + branch, tests written first, `ruff check` clean, CLAUDE.md
files updated, learnings captured as `.claude/skills/`.

- **Phase 0 — Scaffold (this change):** design doc, CLAUDE.md files, repo skeleton.
- **Phase 1 — Backend core:** Poetry env, config, storage layer, auth (register/login/me with
  bcrypt + signed cookies), deck CRUD with stub cards. *Tests: auth flows, deck CRUD, storage
  atomicity, per-user isolation (user A cannot read user B's decks).*
- **Phase 2 — Generation:** anthropic integration, skill prompt, structured output, upload
  endpoint end-to-end. *Tests: mock the Anthropic client; validate schema handling, size
  limits, error paths. One live smoke test behind an env flag.*
- **Phase 3 — Frontend:** Vite + Tailwind scaffold, login screen, Upload tab (file +
  name/description + progress state), Decks tab (list → deck view → study mode: shuffled
  flip-through with "again / got it", missed cards recycle), card/deck edit + delete.
- **Phase 4 — Polish:** nicer error/loading states, mobile-friendly layout, empty states.
- **Phase 5 (later, pre-public):** background generation with status polling, SQLite
  migration, spaced repetition (SM-2), rate limiting + upload caps, Claude Agent SDK if
  generation becomes multi-step/agentic.

## MCP server recommendations

Claude Code's built-ins (bash, file tools, web search) cover most of this project. Two are
actually worth installing:

1. **Playwright MCP** (`npx @playwright/mcp@latest`) — lets Claude drive the running React app
   in a real browser to verify UI flows end-to-end (login → upload → study). This is the
   highest-value one for Phase 3+.
   `claude mcp add playwright -- npx @playwright/mcp@latest`
2. **Context7** (optional) — up-to-date library docs on demand; useful if we hit fast-moving
   FastAPI/Vite API surface questions.

Skip for now: GitHub MCP (the `gh` CLI already covers PRs/issues), database MCPs (no DB yet),
filesystem MCPs (built-in).

## Decisions log — dynamic quiz (interview, 2026-07-05)

| Decision | Choice |
|---|---|
| Topic content source | PDF upload → Claude extracts a markdown notes doc at creation; the quiz runs off the notes doc, corrections edit it, original PDF kept (downloadable) |
| Question generation | One at a time, adaptive: each call sees the notes, the mastery memory, and the session so far |
| Mastery score (0-100) | Agent-decided at session end, judged over the whole topic; **can decrease**; starts at 0 |
| Persistence | Compact session summaries only — full chat transcripts are ephemeral (active session state survives reloads until finished/abandoned) |
| Dispute flow | Agent rules on the objection: may raise the grade (never lower), may append a correction to the notes doc, always recorded for future sessions |
| Visibility | Notes doc and the tutor's "why this score" memory are both user-visible (Notes & progress tab) |
| Question count | Free number input, 1–25 |
| Latency | Synchronous calls with a thinking indicator (matches deck generation); ~2-6s per turn |
| Multi-tab safety | `start` requires `replace:true` over an active session; `answer`/`dispute` carry a session/question binding → 409 for stale tabs |
